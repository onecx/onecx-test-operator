# AGENTS.md — onecx-test-operator

## 1. Technology Stack

| Component | Version |
|---|---|
| Quarkus | 3.x (parent `onecx-quarkus3-parent:2.4.0`) |
| Java | 17 |
| Maven | ≥ 3.8 |
| Docker | required for native build and container image |

Key libraries: Fabric8 Kubernetes Client, Vert.x WebClient, MapStruct, Lombok,
MicroProfile OpenAPI, Micrometer/Prometheus, OpenTelemetry, tkit-quarkus-* libs.

## 2. Build and Test Commands

```bash
# Build with unit tests
mvn package

# Run integration tests
mvn verify

# Clean build
mvn clean package

# Run a specific test method
mvn test -Dtest=<TestClass>#<testMethod>
```

> The `initialize` Maven phase downloads `onecx-test-operator-v1.yaml` from
> GitHub into `target/tmp/openapi/`. An internet connection is required for
> a fresh build.

## 3. Coding Standards

- Use `jakarta.*` imports — never `javax.*`.
- Use **MapStruct** for all mappings between REST DTOs and domain objects.
  Mappers live in `operator/rs/v1/mappers/` and `operator/ui/mappers/`.
- Use **Lombok** where appropriate to reduce boilerplate.
- **OpenAPI-first** for REST controllers: server stub interfaces are generated
  by `org.openapitools:openapi-generator-maven-plugin` (jaxrs-spec / quarkus).
  Controllers implement the generated `*ApiService` interface — never hand-write
  JAX-RS annotations on controllers.
- **REST clients** are generated by the quarkiverse codegen integrated plugin
  (`quarkus-openapi-generator`). Place client OpenAPI specs in
  `src/main/openapi/` and reference them via `application.properties`.
- Prefer **field injection** (`@Inject`) over constructor injection.
- Use `@ApplicationScoped` as the default CDI scope for services and controllers.
- Use `tkit-quarkus-log-cdi` / `tkit-quarkus-log-rs` for structured logging —
  do not add raw `System.out` or `java.util.logging` calls.
- Use modern `java.time.*` API — never `java.util.Date` or `java.util.Calendar`.

## 4. Do Not Rules

- **Never add dependencies** outside the Quarkus ecosystem or `org.tkit.*`
  without explicit user approval.
- **Never set dependency versions** in this child POM — versions are managed
  by the parent (`onecx-quarkus3-parent`).
- Do **not** use `java.util.Date` or `java.util.Calendar`; use `java.time.*`.
- When implementing tests for **existing functionality**, do NOT change the
  implementation just to make a test pass. If you suspect a bug, ask the user
  whether to fix the bug or adjust the test.
- Do not bypass `@QuarkusTest` lifecycle — never start the app manually inside
  tests.

## 5. Testing Standards

- All REST controller tests are `@QuarkusTest` integration tests that exercise
  the full HTTP layer using **RestAssured**.
- Test infrastructure:
  - `AbstractTest` — base class; provides `MockServerClient`, `KubernetesClient`,
    `KeycloakTestClient`, and RestAssured Jackson config.
  - **MockServer** (`quarkiverse-mockserver`) — stubs external HTTP calls
    (e.g., permission service).
  - **Keycloak dev service** (`quarkus-test-keycloak-server`) — provides real
    OIDC tokens; use `KeycloakTestClient` to obtain tokens.
  - **`@WithKubernetesTestServer`** — provides a local Kubernetes mock API.
  - **`@InjectMock` + Mockito** — mock CDI beans that cannot be covered by the
    Kubernetes mock (e.g., `K8sExecService` which uses Vert.x WebSocket exec).
  - **`@GenerateKeycloakClient`** — declares OIDC test clients with scopes
    (from `tkit-quarkus-security-test`).
- For service-layer logic, use `@QuarkusTest` with `@Inject` (see
  `TestServiceToJsonTest`) — only reach for `@InjectMock`/Mockito when you
  need to isolate an unrelated collaborator.
- Annotate each controller test class with `@TestHTTPEndpoint(<Controller>.class)`
  to avoid hardcoding paths.
